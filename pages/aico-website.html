AICO needed to modernize their integrated website platform. My analysis and research spanned integration mapping, top-down stakeholder interviews, performance testing, and surveys across multiple user groups.

The technical challenge involved designing integration patterns between outdated ASP-based code, the modernized code, and the backend system. I designed SOPs and architectural documentation for future updates and rebuilt the experience using modern design tools. With AI assistance, I custom engineered a PHP application to bridge the gap between contemporary design systems and legacy infrastructure. This enabled design access for non-technical users while maintaining security requirements.

The final product enabled integration endpoints with furniture tech, easy internal design access, and a responsive layout across 7 breakpoints. Client facing, it featured a threefold user flow that funnels browsing consumers while simplifying information gathering for sales reps, retailers, and internal users.

Project Summary

AICO needed to modernize their website with a new UI that could adapt to emerging furniture industry technology. Managing thousands of SKUs through a backend server meant any updates required the same server-dependent process, creating operational constraints from the start.

As product manager and designer, I collaborated across IT, Marketing, and Executive departments, applying interpersonal, coding, analytical, and UX skills to navigate constraints and deliver a complete solution. My research and analysis spanned observational studies, top-down stakeholder interviews, A/B testing, and surveys across every persona group. I presented these findings to secure executive approval for UX-specific updates from leadership, including the President and CEO.

The technical challenge involved untangling outdated ASP-based code from the backend system. I created SOPs and signposts for future updates, then rebuilt the experience using Webflow designs. To bridge the gap between modern design tools and legacy infrastructure, I custom built a Webflow-AICO Server PHP application with AI assistance, enabling the integration while maintaining system requirements.

The final product delivered a complete UI transformation with responsive design across 7 breakpoints. It featured a threefold user flow that funnels browsing consumers while simplifying information gathering for sales reps, retailers, and internal users.

What This Project Solved

The Main Problems

  • Interface Update: CEO Michael Amini was focused on aesthetics. He wanted the client facing site and internal portal to reflect AICO's awe-inspiring showrooms, like their High Point location featuring a rebuilt Trevi Fountain.
  • Limited Budget: AICO received quotes between $100-350k for this project but couldn't justify the cost during a slow business period, despite needing a comprehensive platform update.
  • Responsive Layouts: The legacy site interface merely scaled content linearly across devices causing a 48% mobile engagement drop, so the new platform needed proper responsive design with optimized layouts at specific breakpoints.
  • Multi-User Architecture: The new platform had to serve three distinct user types—internal staff, sales reps needing instant access, and browsing consumers—under one unified architecture without confusing overlap.
  • Marketing Independence: The Marketing department designed campaigns but couldn't update the platform independently, creating operational bottlenecks with IT that slowed new campaign launches.
  • IT Security Protocols: IT had to handle all updates due to their server-first policy and concerns that non-technical employees could compromise the security of the complex, integrated platform.
  • Backend & Server Integration: A combination of IT's and Marketing's requirements, marketing needed the ability to update the platform with minimal technical skills while ensuring changes met IT's server and security requirements.

The Journey

There were five distinct phases: initial stakeholder alignment and constraint mapping, comprehensive user research across multiple personas, technical architecture development bridging incompatible systems, integrated design implementation balancing competing requirements, and systematic deployment with measurable organizational impact.

Phase 1

Stakeholder Alignment & Constraint Mapping

Initial engagement with executive leadership established the scope of organizational transformation required beyond standard platform redesign.

Comprehensive constraint mapping revealed the intersection of budget limitations, security requirements, and operational independence needs across departments.

Phase 2

Multi-User Research

Investigation across three distinct user groups revealed a misalignment between system architecture and actual workflow patterns.

Research methodology included operational studies, structured interviews, and behavioral analysis to identify root organizational inefficiencies.

Phase 3

Architecture Development

Development of custom bridging program enabled  connection between modern design tools and legacy security infrastructure.

This technical innovation resolved the fundamental incompatibility between visual design requirements and operational constraints.

Phase 4

Unified Architecture Implementation

Implementation of context-sensitive interface systems accommodated diverse user needs without compromising luxury brand positioning or operational efficiency.

Architectural decisions integrated multi-user needs with executive aesthetic requirements through an adaptive navigation design.

Phase 5

Deployment and Optimization

Rollout included comprehensive training, performance monitoring, and iterative refinement based on user feedback.

Results demonstrated successful transformation of organizational workflows while maintaining all original constraint parameters.

The Data

01
24.6 Second 81% Bounce Rate Homepage Problem
X

Performance Metrics:

  • Homepage engagement time: 24.6 seconds
  • Total monthly homepage sessions: 23,435
  • Homepage bounce rate: 81%

The platform’s primary entry point exhibited high abandonment rates, with users terminating sessions immediately without accessing further content or core service endpoints.

Business Impact:

Users found the platform’s entry point but couldn't determine next steps or value propositions within 25 seconds. This created a conversion bottleneck at the most critical touchpoint, wasting the majority of organic traffic acquisition efforts.

02
39,647 Views on One Single Template
X

Core Platform Usage Patterns

The /collections_detail template generated 39,647 monthly views—more traffic than any other section. Users skip general browsing and go directly to specific product endpoints, making individual “product showcase” the dominant feature.

Resource Allocation Decision

Since one template serves the majority of user interactions, optimizing the product detail page would impact the largest possible portion of platform traffic. This data justified concentrating design resources on perfecting the product showcase rather than spreading efforts across multiple page types.

03
48% Overall Traffic, 48% Less Mobile Engagement
X

The Mobile Problem

  • Mobile client sessions: 48% of all traffic
  • Desktop session duration: 199 seconds
  • Mobile session duration: 104 seconds (48% lower than desktop)

Nearly half the user base experiences significantly degraded system performance, directly impacting business outcomes.

Linear Scaling Failure

The old platform scaled content linearly across devices instead of implementing responsive architecture. This technical approach created major usability issues for the substantial mobile audience seeking product information.

04
53% Organic Traffic, 3,638 High-Intent Dealer Searches
X

Traffic Source Concentration

Organic search generates 53% of all sessions, making it the primary traffic source. Heavy reliance on search means system performance issues could jeopardize the main channel bringing users to the platform.

Purchase Intent Signal

The /locator endpoint received 3,638 views—indicating strong "purchase-ready" behavior from users seeking physical location data. This became key data for executive strategy discussions and e-commerce platform planning, demonstrating clear user intent to transition from browsing to transaction completion.

05
68.5% Engagement Rate Masks Deeper Issues
X

Surface-Level Metrics

  • Organic search engagement rate: 68.5%
  • Average session duration: 141.7 seconds

These numbers comprise every user type and any engagement type from every entry point to the website. While these numbers appear reasonable, one-third of users terminate sessions without meaningful interaction.

Quality vs. Quantity Problem

The disparity between average session duration and productive interactions revealed that users successfully find the platform but a large portion fail to convert visits into business-relevant activities. This indicated systematic issues with platform flow architecture rather than traffic acquisition problems.

The Website After:

Reflections

What I learned and What I would do Differently
What I Learned

I had the opportunity to combine a lot of different types of tasks in this project and I enjoyed it precisely because it was so multifaceted. One day I was conducting research and proving architectural concepts, another day aligning departments, and the next getting lost in brainstorming new solutions or coding.

But I also learned that I can get carried away when given creative freedom. I adored working more on the project rather than in it—designing integration patterns and building those into our server code iteratively. Yet this autonomy made me assume I knew what was best more often than I actually did. I realized I was working on the project in isolation when I should have been collaborating more. I should have asked colleagues for input first, rather than rolling out what I thought was best across the board.

I especially learned to be firm with executives on decisions they might see as “deviations”. When I realized the integration of interface updates with the backend server should be automated, I took it to the execs. Naturally, a process hidden from all except myself and a couple colleagues prompted push back. I pointed to my analysis detailing the incongruence between marketing and IT, proving it was worthwhile. Luckily, they assented, and now that “deviation” gives marketing design control, and saves days in delays between them and IT.

What I Would do Differently

I would have done far more stakeholder validation throughout rather than being so insular. There were many small instances where I made design decisions that I later had to refactor—like adding a mask over product images that made them look too dark, or adjusting font sizing that everyone disliked, or adding a broken underline under main titles when people preferred one solid line. These things were small, and usually interface-related, but they added up and could have been avoided if requested feedback first.

I also would have reviewed third-party service agreements before building the integration application. I was set back about one week when I discovered my web-scraping approach violated terms. I had to transform the whole application into a new version that handled file processing instead. I wish I took the time to check third party compliance requirements first, instead of storming ahead and costing time.

Essentially, I should not build and implement without feedback in the future. I should have collaborated much more from the start rather than having to painstakingly roll back updates. I don’t always know what is best, even when I think I do.
